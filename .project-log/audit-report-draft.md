### 实测验证有效（做得好）
- **核心流水线优秀（9/10）**：claim → 同步视频 → 指标 → 提交 → 自动跳转 → 自动释放旧锁 → 循环。toast 清晰，`remainingCount` 正确递减，URL 逐条推进，Pass/Fail 多轮验证通过。

- **提交校验**：Fail 缺主原因码客户端拦截；主原因码字段仅 Fail 出现；2400 字备注可接受；revisions 正确持久化。
- **权限路由**：`/dashboard` `/task-types` `/accounts` `/database` 均重定向回 `/reviewer`；登出清 session。
- **指标重排**：正确 bad→warn→good；help tooltip 中文清晰。

### 质检员体验评分：**7 / 10**

## 二、管理员（admin / qc_manager）视角审计

**测试账号：** admin / Admin123!

### 正确性发现（Hat A）

| 严重度 | 问题 | 影响 |
|--------|------|------|
| 🔴 HIGH | **派发模式切换静默重置抽检% 到 100%（过量派发）** | 工作台派发工作区：选批次→百分比抽检设 25%→Generate（待派发=1 ✓）→点"全量派发"→再点"百分比抽检"→**不重新输入直接 Generate**，百分比字段静默回弹到 100%，生成整批（5/5）。纯采样改值（25%→60%→25%）正常，**仅 full→sampled 切换会丢值**。管理员回到采样想重派，一点就把整批派出去且无任何警告。最高频关键操作。 |
| 🔴 HIGH | **任务状态机矛盾态 `assigned + fail`（数据黑洞）** | `batch_813ae812ad867e8c` 中 1 条 episode `qcStatus:assigned` 却 `qcResult:fail`。该任务从 reviewer 的待办/进行中/已完成/总分配全部计数中消失。疑似 submit 提交 fail 后被某次 dispatch 重派发把 `qcStatus` 拉回 assigned 却残留 `qcResult=fail`，即 submit 与 dispatch 重置存在覆盖/竞态漏洞。**这是质检员"看板隐藏任务"的根因。** |
| 🟡 MEDIUM | **"删除"任务类型其实是软停用，无硬删除** | 删除任务类型 toast 提示"已删除"，但刷新后类型仍在目录里、只是翻成"已停用"（6 次删除均如此）。批次正确回收到待分类，但类型永远残留。这正是历史垃圾（`A&B test`、`<script>x</script>` 等）清不掉、越堆越多的原因。 |
| 🟡 MEDIUM | **跨页批次/集数对不上（幻影 +1 批次）** | Database 显示 47 批次/4716 集；QC历史总览显示 48/4717；待分类详情标签写"5 批次·304 集"但表格只渲染 4/303；QC历史范围选择器列出 47 个可选批次但卡片写 48。一个含 1 集的幽灵批次被某些聚合计数、却不在任何实际列表里。管理员无法信任任何总数。 |

## 三、合并结论与修复优先级

### A. 正确性修复清单

**🔴 必须优先（HIGH）**
1. **派发模式 full→sampled 切换静默重置抽检% 到 100%** → 过量派发整批。保留上次输入的%，或生成前显示有效样本量。最高频关键操作，最易误伤运营。
2. **任务状态机矛盾态 `assigned + fail` 数据黑洞**（=质检员"看板隐藏任务"根因）。排查 submit 与 dispatch 重派发的状态覆盖：① 已 done 任务重派发不得被拉回 assigned；② 禁止 qcStatus/qcResult 矛盾组合。
3. **根路径 `/` 白屏崩溃**（Vue Router 无限重定向栈溢出）。修 `beforeEach` 对裸 `/` 的重定向。

**🟡 应尽快（MEDIUM）**
4. **`/api/task-pool` 对 reviewer 越权返回全量**。后端按角色裁剪 payload，不靠前端表格过滤。
5. **"删除"任务类型/账号实为软停用且无硬删除** → 垃圾无法清理。提供真删除，或老老实实改文案为"停用/归档"。
6. **跨页批次/集数不一致（幻影 +1 批次）** + **任务类型批次计数 stale**。统一计数口径与事实源，排查幽灵批次来源。
7. **离开质检页不释放锁 + 管理员无强制回收锁入口**。reviewer 端导航离开释放锁，管理员端补回收兜底。
8. **派发可发现性**：工作台显示"批次未派发(new)数量"。
9. **质量评分环显示不一致指标**：固定为稳定的综合 Q_motion。
10. **任务类型名输入未校验**：加服务端校验/清洗（当前仅靠转义侥幸无 XSS）。

### B. 体验优化 backlog（按 频率×痛感 排）
- 登录错误展示原始 JSON（high）→ 只渲染 detail
- "删除"文案撒谎（high）→ 改"停用/归档"或真删
- 原因码英文未汉化（high，质检员高频）→ 中文映射
- 默认空看板首屏（medium）→ 默认选有活的类型 / 加"有活"标记
- "返回总库"标签与去向不符（medium）
- 下载存成 `.bin`（medium）→ 按 mimeType 给扩展名
- 确认框英文 Cancel/OK、确认派发过早可点、无客户端登录校验、无媒体集播放控件可点、QC历史表不分页（low）

### C. 覆盖度声明
- **浏览器实测**：质检员全流程（登录/看板/任务池/质检流水线/视频/提交/权限）+ 管理员全页面（登录/派发工作区/任务类型/数据总库/账号/QC历史/导航）。
- **API 层补充**：管理员侧数据一致性、越权、状态机矛盾（挖到浏览器点击看不见的 `assigned+fail` 黑洞）。
- **未覆盖**：庆祝动画最终遮罩（需做完全部任务，会污染数据）；跨用户锁冲突（reviewer02/03 密码未知）；管理员实际提交 QC（按不批量提交约定跳过）；派发 dispatch-assign 平均/自定义的大规模写流程（仅小批次验证，建议隔离环境专项）。

### 总体评价
核心质检流水线工程质量高（质检员 7/10，纯审核循环 9/10）。最大风险集中在三类：**派发可靠性**（抽检%静默重置→过量派发）、**状态机/计数一致性**（assigned+fail 黑洞、幽灵批次、stale 计数）、**入口与可发现性**（根路径崩溃、看板待办不全、空首屏）。这些会在用户触及优秀核心功能前就造成误操作、数据遗漏或困惑，应优先于体验优化处理。
