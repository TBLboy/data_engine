## 二、项目背景

### 2.1 这是什么

**Robot QC V1** 是一个公司内网使用的机器人遥操作数据质量检测平台。

核心业务：采集人员操作机器人产生 episode → 自动扫描入库 → L3 引擎自动算分 → 质检员在 Web 页面逐条审核（看视频 + 看指标 + 提交 pass/fail）。

### 2.2 数据流

```
MinIO 对象存储（原始数据）
  → 后端扫描入库（PostgreSQL）
    → L3 v2 引擎自动计算质量指标
      → 质检员在 manual QC Web 页面审核
        → 提交 QC 结论（pass/fail + 原因码）
```

### 2.3 机器人硬件

- 机械臂：双臂 robot，单臂 7 DOF（joint position）+ 灵巧手（position 控制，值域 0-255）
- 传感器：三路 RGB 相机 + 深度相机
- 数据格式：每 episode 包含 telemetry.npz（qpos/actions/effort/qvel/时间戳）+ 视频 MP4
- 采集频率：通常 30 fps，一集约 20-60 秒

### 2.4 技术栈

| 层 | 技术 |
|----|------|
| 后端 | Python 3.x + FastAPI |
| 数据库 | PostgreSQL 17 |
| 存储 | MinIO（S3 兼容对象存储） |
| 前端 | Vue 3 + Element Plus + Chart.js |
| 部署 | Docker Compose（frontend / backend / db 三容器） |
| 运行环境 | Linux 内网服务器，**无 GPU** |

### 2.5 项目阶段

当前处于 V1 生产调优阶段，已在公司内网实际使用。L3 指标体系刚完成对 lerobot-doctor（开源遥操作数据质量工具）的全面对标审查，11 条检查项已闭环。

## 三、L3 v2 引擎详情

### 3.1 四层架构

```
FeatureExtractor（特征层）
  → MetricEngine（指标层）
    → QualityEngine（融合层）
      → 前端展示层
```

### 3.2 12 项指标总览

#### MQ — Motion Quality（运动质量，权重 0.40）

| 指标 ID | 名称 | 含义 | 算法 |
|---------|------|------|------|
| MQ-01 | 平滑度 | 关节运动是否平滑 | 三阶 jerk（加加速度）RMS，越低越平滑 |
| MQ-02 | 动作饱和 | 关节是否频繁顶到机械限位 | qpos 长时间贴近 min/max 边界 |
| MQ-03 | 无效动作 | 是否做无意义的抖动 | 高频低幅振荡检测 |

#### LQ — Learning Quality（学习质量，权重 0.40）

| 指标 ID | 名称 | 含义 | 算法 |
|---------|------|------|------|
| LQ-01 | 动作密度 | 有效动作占比 | 排除静止/微动帧后的帧数占比 |
| LQ-02 | 停滞检测 | 长时间无动作 | 连续 frames 无显著位置变化 |
| LQ-03 | 跟踪误差 | action 是否被准确执行 | action 目标位置 vs 实际 qpos 的 RMS error，含滞后对齐 |

#### DI — Data Integrity（数据完整性，权重 0.20）

| 指标 ID | 名称 | 含义 | 算法 |
|---------|------|------|------|
| DI-01 | 时间戳规则性 | 是否有丢帧/时间戳异常 | 声明 FPS → 预期帧间隔，检测 >1.5 倍的丢帧 + 抖动 + FPS 一致性偏差 |
| DI-02 | 传感器同步 | 多模态传感器是否对齐 | 深度相机 vs 关节数据时间戳自适应阈值对齐检测 |
| DI-03 | 维度健康 | 传感器是否故障 | 逐维度零方差检测（std==0）+ MAD robust z-score 异常值检测 |
| DI-04 | Episode 完整性 | 数据是否完整 | 帧数过短推断（<10 frames）+ manifest 声明 vs 实际帧数差异（>5%） |

#### DX — Execution Diagnostics（执行诊断，不参与总分）

| 指标 ID | 名称 | 含义 | 算法 |
|---------|------|------|------|
| DX-01 | 执行跟踪 | action→qpos 逐帧跟踪 | 加权 RMS error，含滞后时间对齐 |
| DX-02 | 数据有效性 | NaN/Inf 扫描 | 扫描 actions/qpos/effort/qvel 四数组 |

### 3.3 综合评分

```
Q_motion = MQ_score × 0.40 + LQ_score × 0.40 + DI_score × 0.20

截断规则：任一 DI 指标 < 3.0 → 总分封顶 3.0
          任一 DI 指标 < 5.0 → 总分封顶 6.0
```

设计意图：数据完整性是地基，地基有问题时，即使运动质量看起来不错也不能给高分。

### 3.4 每项指标的输出结构

```python
{
    "metricId": "DI-01",
    "name": "时间戳规则性",
    "score": 3.2,           # 0.0 - 10.0
    "level": "warn",         # good / warn / bad
    "value": 3.2,
    "valueText": "3.2 分",
    "unit": "分",
    "description": "第 3.2-5.1 秒存在 8 帧丢失，FPS 一致性偏差 12%",
    "qualityDimension": "data_integrity",
    "timelineSegments": [    # 异常时间线片段
        {
            "startSec": 3.2,
            "endSec": 5.1,
            "label": "丢帧",
            "level": "warn",
            "sourceMetricId": "DI-01"
        }
    ]
}
```

### 3.5 DI 指标对 Q_motion 的截断示例

一段实际数据的评分：

| 指标 | 分数 |
|------|------|
| MQ-01 | 8.5 |
| MQ-02 | 9.0 |
| MQ-03 | 7.2 |
| LQ-01 | 8.0 |
| LQ-02 | 6.5 |
| LQ-03 | 7.8 |
| DI-01 | 8.0 |
| DI-02 | **1.6** |
| DI-03 | 10.0 |
| DI-04 | 10.0 |

不加截断：MQ=8.2, LQ=7.4, DI=7.4 → 加权 = 7.7 分（良好）

加截断后：DI-02 = 1.6 < 3.0 → **总分封顶 3.0 分**（差）

质检员如果不理解截断逻辑，看到 7.7 变 3.0 会很困惑 — **这正是需要 AI 解释的场景**。

### 4.2 当前痛点

1. **信息密度高但缺乏整合**：质检员看到 12 个数字 + 多条曲线 + 异常段标签，需要自己在脑中拼出"这段数据到底有什么问题"
2. **截断逻辑不透明**：DI-02 1.6 分把总分从 7.7 拉到 3.0，质检员不理解为什么要降这么多
3. **异常段与指标的关联不明显**：timeline 上标了红色/黄色段，但每个段是哪个指标触发的、严重程度如何，需要质检员手动对照
4. **灵巧手指标容易被忽略**：当前 12 个指标中灵巧手相关的不突出，质检员可能漏看手部数据问题

## 五、拟议方案

### 5.1 总体策略：两阶段推进

#### 第一阶段：结果解释（result interpretation）

- **输入**：结构化的 L3 指标 JSON（12 个指标的 score/level/description/timeline）
- **输出**：自然语言质检总结
- **模型能力要求**：结构化数据 → 自然语言翻译 + 简单逻辑推理（"因为 DI-02 很低所以触发了截断"）
- **不动现有业务逻辑**：只是在前端把已有指标数据多渲染一个"AI 解读"卡片

#### 第二阶段：辅助检查（assisted inspection）

- **输入**：遥操作曲线图截图 + 维度指标数据
- **输出**：是否存在漏检异常段、是否存在疑似伪影
- **模型能力要求**：多模态（看图 + 读数值）+ 遥操作异常模式识别
- **这一阶段难度跃升**，需要先建立评估基准

### 5.2 第一阶段架构草稿

```
前端 manual-qc.vue
  │
  ├─ 已有: fetchManualQcContext() → L3 指标 JSON
  │
  └─ 新增: fetchAiExplanation(metricJson)
            │
            ▼
         后端 /api/ai/explain
            │
            ├─ 构建 prompt 模板（填入指标数据）
            │
            ├─ 调用 Ollama / vLLM API（本地）
            │    model: qwen2.5:7b 或 deepseek-r1:8b
            │
            └─ 返回 { explanation: "...", model: "...", latencyMs: ... }
            │
            ▼
         前端渲染 "AI 解读" 卡片
```

### 5.3 Prompt 模板（草稿）

```
你是一个机器人遥操作数据质量分析助手。

以下是一个 episode 的 L3 自动质检结果。请用中文为质检员写一段简洁的解读报告。

要求：
1. 先概括数据整体质量（好/需要关注/较差）
2. 指出最突出的 1-2 个问题（score < 5 的指标优先），解释可能原因
3. 说明 Q_motion 评分是否受到截断规则影响
4. 如果所有指标正常，简洁说明即可，不要编造问题
5. 控制在 200 字以内

指标数据：
<此处填入 JSON>

Q_motion 截断规则：
- 任一 DI 指标 < 3.0 → 总分封顶 3.0
- 任一 DI 指标 < 5.0 → 总分封顶 6.0

当前 Q_motion 分数：{score}/10，等级：{level}

请开始解读：
```

## 六、环境与约束

### 6.1 服务器配置

- OS：Ubuntu 22.04.5 LTS
- CPU：32 cores
- RAM：31 GB（当前空闲约 21 GB）
- GPU：**NVIDIA GeForce RTX 4090 × 1，24 GB VRAM**
- CUDA：13.0，Driver：580.159.03
- GPU 当前占用：~555 MB / 24 GB（大量空闲）
- Python：3.10.20（项目 conda 环境），python:3.10-slim（Docker 镜像）
- 部署方式：Docker Compose v2.40.3 + Docker v29.1.3
- Ollama：尚未安装
- NVIDIA Container Toolkit：状态未知（Docker GPU 透传未验证）

### 6.2 约束

- **必须本地部署**：数据不能出内网，不能用 OpenAI/Claude API
- **有 GPU**：RTX 4090 24G，足以流畅运行 7B-14B 模型，甚至可跑 32B 量化模型
- **不能修改现有 L3 指标计算逻辑**：AI 只做"解读"，不做"改分"
- **不能阻塞质检流程**：AI 解读是可选的辅助信息，加载失败不影响提交 QC 结论
- **必须支持中文**：质检员和系统界面都是中文

### 6.3 非功能需求

- 解读延迟目标：< 2s（RTX 4090 上 7B 模型生成 200 字中文约 1-2s）
- 错误降级：模型不可用时静默 fallback，不阻塞页面
- 并发：当前质检员数量少（个位数），无需考虑高并发
