## 一、平台定位

Robot QC V1 是面向公司内部机器人遥操作数据生产环境的**训练数据资产管理与质量管控系统**。核心职责是确保数据湖中的海量原始数据经过自动与人工结合的质检筛选后，持续产出**可直接用于模型训练的高质量数据资产**。

### 核心目标

- **数据管理**：将 MinIO 数据湖中的原始数据规范化入库，按批次管理，按任务类型分类
- **数据质检**：通过四层 QC 体系（L1-L4）对每批数据进行质量评估
- **数据资产供给**：仅质检合格的批次和 episode 进入训练集，对接下游模型训练管线

## 五、核心功能详解

### 5.1 数据全生命周期管理

1. **MinIO 数据湖全量扫描入库**
   - 递归遍历 `yaocao` bucket，通过结构特征识别 list 和 episode
   - 幂等落库，增量同步
   - 将 `ingestable → processable → qc_ready` 三层状态写入 PostgreSQL

2. **批次管理**
   - 批次是业务最小管理单元，具有全生命周期状态：
     - `待分类 → 已分类 → 抽检中 → 已完成 → 部分驳回 / 整批驳回`
   - 支持按任务类型分类、改挂、回收

3. **任务类型管理**
   - 人工维护的业务目录（admin/qc_manager）
   - 扫描器提供自动分类建议（authoritative + suggest-only seed 规则）
   - unclassified/batch 回收语义完善

4. **多角色权限**
   - **admin**: 系统管理员，全部操作权限
   - **qc_manager**: QC 管理，派发/巡检/报表/驳回
   - **reviewer**: 质检员，个人看板 + 任务池 + 质检流水线
   - **viewer**: 只读，查看概览与报表

### 5.2 人工质检（Manual QC）

生产级人工质检流水线，支持多质检员并行协作：

- **任务派发**：工作台选择批次 → 生成待派发任务池 → 批量派发 reviewer
- **认领/锁定/提交**：reviewer 认领任务锁定，防止多人冲突
- **流水线模式**：提交后自动跳转下一条任务
- **Double 盲审**：多人独立评审，聚合结果判定
- **庆祝动画**：全部任务完成后触发礼花彩条 + 音效
- **重新质检**：reviewer 可申请，admin 可接管 reopen

**人工审核内容**（L2 + L4）：
- L2 视觉质量：模糊、过曝、遮挡、抖动、深度异常、动作流程完整性
- L4 任务完成度：episode 是否完成预设任务目标

### 5.3 L3 自动指标评估

基于 telemetry.npz 数据，程序自动计算 6 项 P0 指标：

| 指标 | 全称 | 检测目标 |
|------|------|---------|
| LDLJ | Large Delta Joint | 相邻帧关节角度跳变过大 |
| Dead Actions | 死区动作 | 多帧无有效动作输入 |
| Action Saturation | 动作饱和 | 关节指令长时间卡在极限值 |
| Static Detection | 停滞检测 | 机械臂长时间无有效运动 |
| Timestamp Regularity | 时间戳规整性 | 帧率抖动、丢帧、时间戳异常 |
| Qpos-Action Tracking Error | 状态-动作跟踪误差 | 实际状态与指令偏差过大 |

P1 增强指标（后续版本）：Per-finger Gripper Chatter（手指颤振）、Joint Effort（关节力矩异常）

### 5.4 批次驳回决策

将平台从"只记录质检结果"升级为"按任务统计训练可用数据、执行驳回策略"：

```
失败率 = 人工失败数 / 抽检数
阈值 θ（默认 0.10，可调整）
→ R_fail > θ → 整批驳回，所有 episode 不可用
→ R_fail ≤ θ → 批次通过，仅人工失败的 episode 不可用
```

### 5.5 数据集管理与导出

- 按任务类型 / 批次 / QC 结果筛选导出
- 仅导出质检合格 episode
- CSV / JSON 清单格式
- 对接下游训练数据消费管线

### 5.6 QC 历史审计

- 审计中间件全量记录所有 QC 操作
- 支持操作追溯和报表
- 管理员任务管理（撤回/转派/释放 pending 任务）

### 5.7 AI 助手

基于本地 Ollama 服务的 AI 质检辅助功能：
- 质检结果解释（AI Explain）
- AI 对话式辅助（AI Assistant Panel）
- 支持自定义 LLM 配置

## 六、数据模型

### 核心表结构

```
User(id, username, name, role, avatar, password_hash, is_active)
  → role: admin / qc_manager / reviewer / viewer

TaskType(id, name, ...)

Batch(id, task_type_id, name, episode_count, qc_status,
       sampling_ratio, pass_rate, failure_rate, reject_threshold,
       batch_decision, ...)
  → qc_status: new / classified / sampled / completed / partially_rejected / rejected
  → batch_decision: PENDING / ACCEPTED / REJECTED

Episode(id, batch_id, task_name, duration_sec, frame_count,
        qc_status, qc_result, reviewer, reason_code,
        manual_qc_status, final_dataset_status, is_exportable, ...)
  → qc_status: new / in_review / done
  → qc_result: pending / pass / fail
  → manual_qc_status: NOT_REVIEWED / MANUAL_PASS / MANUAL_FAIL
  → final_dataset_status: PENDING / QUALIFIED / UNQUALIFIED

QcTask(episode_id, batch_id, assignee, reviewer_id, status,
        result, lock_owner, lock_expires_at, version, is_active, ...)
  → status: new / assigned / in_review / done

QcReviewRevision(episode_id, reviewer_id, qc_result, reason_code,
                  final_decision_source, submitted_at, ...)

BatchDecisionLog(batch_id, decision, failure_rate, reject_threshold, ...)

DatasetExportJob(id, task_type_id, batch_id_filter, created_by, status, ...)
```

### Episode 最终判定逻辑

| 批次判定 | 人工状态 | 最终状态 | 说明 |
|---------|---------|---------|------|
| REJECTED | MANUAL_FAIL | UNQUALIFIED | 人工失败 |
| REJECTED | MANUAL_PASS | UNQUALIFIED | 被批次驳回覆盖 |
| REJECTED | NOT_REVIEWED | UNQUALIFIED | 批次驳回传播 |
| ACCEPTED | MANUAL_FAIL | UNQUALIFIED | 人工失败 |
| ACCEPTED | MANUAL_PASS | QUALIFIED | 可用 |
| ACCEPTED | NOT_REVIEWED | QUALIFIED | 批次推断通过 |
