## 二、v1 → v2 核心修改

### 修改 1：`skip_until_next_change` → `next_scan_at`

**v1 方案（有缺陷）：**
```
连续 3 次无变化 → skip_until_next_change = TRUE
增量扫描跳过 → 等待下次 full scan 才重新扫
```

问题：设为 TRUE 后，系统无法知道 prefix 何时再次变化（没有 MinIO 事件通知）。

**v2 方案：**
```
每次扫描后计算 next_scan_at：
  刚检测到变化 → now + 30min
  1 次无变化    → now + 2h
  2 次无变化    → now + 12h
  3 次无变化    → now + 1day
  长期无变化    → now + 7days

Worker 领取 shard 时：IF next_scan_at > now() → 跳过

检测到变化 → consecutive_unchanged = 0, next_scan_at = now + 30min
```

`scan_prefix_states` 字段变更：
```sql
-- 删除
skip_until_next_change BOOLEAN

-- 新增
next_scan_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
scan_policy           VARCHAR(32) NOT NULL DEFAULT 'adaptive'
  -- 'adaptive' | 'realtime' | 'daily' | 'weekly' | 'manual'
```

这样即便规模增长也不会有"永久跳过"的盲区，而且天然获得冷热区分效果。

### 修改 2：删除检测收窄到 shard 范围

**v1 方案（有风险）：**
```
全 job 完成 → 标记所有未 seen 对象为 inactive
```

在增量模式下，被跳过的 prefix 的合法对象会被误杀。

**v2 方案：**
```
每个 shard 成功后：
  仅在该 shard 的 prefix 范围内
  对比本次 list_objects 结果与 episode_objects
  存在于 DB 但不在本次 list 结果中的 → 标记 episode_objects.inactive = TRUE
```

以下状态的 shard **不执行**删除判断：`skipped / failed / cancelled / timeout`。

这样即使 51 个 shard 成功、1 个被跳过，被跳过的那个 prefix 内的对象完全不参与删除判断。

### 修改 3：Asset Recompute 竞态修复

**v1 方案（存在竞态）：**
```python
# enqueue_batch_asset_recompute
on_conflict_do_update(..., set_={'status': 'pending', ...})
# 无条件重置 status → 可能覆盖 running
```

**实际代码核实结果：**

当前代码中：
- `mark_recompute_started()` 用 `with_for_update()` 行锁 + `status in ('pending', 'failed')` 检查，运行中的 job 不会被重复领取
- **但** `enqueue_batch_asset_recompute()` 的 `on_conflict_do_update` 无条件把 status 重置为 `pending`，即使 job 正在 running

竞态路径：
```
Worker 处理中 (status=running)
  → scanner enqueue 发到同 batch
  → on_conflict_do_update: status=pending (覆盖!)
  → 调度器下一次轮询取到 (status=pending)
  → Worker 2 也领取 → 重复处理
```

**v2 方案：新增 `rerun_requested`**
```sql
ALTER TABLE batch_asset_recompute_jobs
    ADD COLUMN rerun_requested BOOLEAN NOT NULL DEFAULT FALSE;

ALTER TABLE task_asset_recompute_jobs
    ADD COLUMN rerun_requested BOOLEAN NOT NULL DEFAULT FALSE;
```

修改后的 enqueue 逻辑：
```python
# enqueue_batch_asset_recompute 改为：
on_conflict_do_update(
    index_elements=[BatchAssetRecomputeJob.batch_id],
    set_={
        'reason': normalized_reason,
        'requested_at': now,
        # 仅在非 running 时重置 status
        'status': case(
            (BatchAssetRecomputeJob.status == 'running', BatchAssetRecomputeJob.status),
            else_='pending'
        ),
        # running 时标记需要重跑
        'rerun_requested': case(
            (BatchAssetRecomputeJob.status == 'running', True),
            else_=False
        ),
    },
)
```

修改后的 recompute 完成逻辑：
```python
# recompute_batch_asset_rollup 完成后：
job = db.query(...).filter(batch_id=batch_id).first()
if job and job.rerun_requested:
    job.status = 'pending'
    job.rerun_requested = False
else:
    job.status = 'done'
    job.attempts += 1
```

## 三、最终数据库新增/变更（综合版）

### 新增 `scan_jobs`（v2）

```sql
CREATE TABLE scan_jobs (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    bucket          VARCHAR(128) NOT NULL,
    scan_mode       VARCHAR(32) NOT NULL DEFAULT 'incremental',
        -- 'incremental' | 'full' | 'prefix_list' | 'manual_prefix'
    status          VARCHAR(32) NOT NULL DEFAULT 'pending',
    priority        INTEGER NOT NULL DEFAULT 50,

    total_shards     INTEGER NOT NULL DEFAULT 0,
    succeeded_shards  INTEGER NOT NULL DEFAULT 0,
    failed_shards    INTEGER NOT NULL DEFAULT 0,
    running_shards   INTEGER NOT NULL DEFAULT 0,
    skipped_shards   INTEGER NOT NULL DEFAULT 0,

    triggered_by     VARCHAR(64),
    trigger_source   VARCHAR(32) NOT NULL DEFAULT 'manual',

    heartbeat_at     TIMESTAMPTZ,
    started_at       TIMESTAMPTZ,
    finished_at      TIMESTAMPTZ,
    cancel_requested_at TIMESTAMPTZ,

    error_summary    TEXT,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 新增 `scan_shards`（v2）

```sql
CREATE TABLE scan_shards (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    scan_job_id     BIGINT NOT NULL REFERENCES scan_jobs(id) ON DELETE CASCADE,
    bucket          VARCHAR(128) NOT NULL,
    prefix          VARCHAR(1024) NOT NULL,
    shard_type      VARCHAR(32) NOT NULL DEFAULT 'prefix_scan',

    parent_shard_id BIGINT REFERENCES scan_shards(id),

    status          VARCHAR(32) NOT NULL DEFAULT 'pending',
    priority        INTEGER NOT NULL DEFAULT 50,
    attempt_count   INTEGER NOT NULL DEFAULT 0,
    max_attempts    INTEGER NOT NULL DEFAULT 3,

    lease_owner     VARCHAR(64),
    lease_expires_at TIMESTAMPTZ,
    heartbeat_at    TIMESTAMPTZ,

    objects_seen    INTEGER NOT NULL DEFAULT 0,
    objects_changed  INTEGER NOT NULL DEFAULT 0,
    episodes_seen   INTEGER NOT NULL DEFAULT 0,
    episodes_changed INTEGER NOT NULL DEFAULT 0,

    next_retry_at   TIMESTAMPTZ,
    timeout_sec     INTEGER NOT NULL DEFAULT 600,
    started_at      TIMESTAMPTZ,
    finished_at     TIMESTAMPTZ,

    error_detail    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE(scan_job_id, prefix)
);
```

### 新增 `scan_prefix_states`（v2）


## 四、部署架构：独立 Docker Worker Service

```
docker-compose.yml

services:
  backend:      # FastAPI（不变）
  frontend:     # Nginx（不变）
  db:           # PostgreSQL（不变）

  scan-worker:  # 新增
    build: ../backend
    command: python -m app.services.scan_worker_main
    environment:
      <<: *backend-env
    depends_on:
      db:
        condition: service_healthy
    deploy:
      replicas: 2
    restart: unless-stopped
```

- Worker 通过 `FOR UPDATE SKIP LOCKED` 从 `scan_shards` 领取
- Worker 崩溃 → Docker `restart: unless-stopped` 自动恢复
- Shard 崩溃 → lease 过期后其他 Worker 接管

## 五、实施顺序 v2（10 步）

```
Step 0   基线测试 + Feature Flag
         ├── 记录当前 scanner 行为基线（耗时、对象数、episode 数）
         └── 新增配置项 SCANNER_V2_ENABLED=false

Step 1   数据库模型落地
         ├── Alembic: scan_jobs_legacy, scan_jobs(v2), scan_shards,
         │   scan_prefix_states, rerun_requested 列
         └── 模型代码同步

Step 2   抽离 business_resolver
         ├── 从 scanner.py 中提取 List/Batch/Episode upsert 逻辑
         │   为独立模块 business_resolver.py
         └── 新旧 scanner 共用同一份解析逻辑，减少漂移风险

Step 3   scan_jobs + scan_shards + 单 Worker（Shadow 模式）
         ├── Worker 主进程 + 领取 + 心跳
         ├── Shadow 模式：只统计不写业务表，与旧 scanner 结果对比
         └── 确认新旧对象数/episode 数一致

Step 4   Prefix Discovery + Full Scan（单 Worker）
         ├── 非递归 prefix 发现 → 生成 shard
         └── 流式 list_objects → 对比 episode_objects 指纹 → 批量写入

Step 5   子进程隔离 + Timeout + Lease + Retry
         ├── 每个 shard fork 子进程执行
         ├── MinIO connect/read timeout
         ├── 父进程监控 wall-clock → terminate → kill
         └── 分片级指数退避重试

Step 6   变化检测 + 增量写入
         ├── (object_key, content_hash, size_bytes, last_modified) 四元组对比
         └── 仅变化对象更新 episode_objects + 触发业务解析

Step 7   Asset Recompute 幂等化
         ├── enqueue 逻辑增加 rerun_requested
         ├── recompute 完成逻辑增加 rerun_requested 检查
         └── 并发测试（两个 shard 同时 enqueue 同一 batch）

Step 8   next_scan_at 自适应调度 + 安全删除
         ├── 实现按 next_scan_at 跳过
         ├── 删除检测限定在成功 shard 的 prefix 范围内
         └── 周期性 full reconcile（scan_mode='full'）

Step 9   API + 前端
         ├── POST /api/database/scan { mode: "incremental" | "full" }
         ├── GET /api/database/scan/status（shard 进度）
         ├── 前端：总分片/成功/失败/运行/等待、取消、重试失败
         └── 删除旧的 scan_queue 代码

Step 10  灰度切换 + 多 Worker 并行
         ├── Feature flag 打开 → 新 scanner 正式写入
         ├── Prefix 灰度：先少量 prefix，确认无误后全量
         └── Worker replicas 从 1 → 2
```

## 七、最终架构一句话

> **Prefix 分片 + PostgreSQL 持久化队列 + 独立 Docker Worker Service + 子进程隔离 + 流式指纹对比增量检测 + `next_scan_at` 自适应退避 + 安全 shard 级删除判断 + 含 `rerun_requested` 幂等资产投影联动 + 前端进度可观测。**

## 八、不做清单（最终确认）

| 不做 | 替代 |
|------|------|
| `object_inventory` 新表 | 复用 `episode_objects`（对象始终归属 Episode） |
| HOT/WARM/COLD/ARCHIVED 四级调度 | `next_scan_at` 自适应退避 |
| 超大 Batch 二级拆分 | 留 `parent_shard_id` 接口 |
| Historical Reconcile 独立表 | `scan_mode='full'` 同一机制 |
| 加权负载均衡 | `priority DESC, objects_seen DESC` 排序 |
